# Board sweeps (cross-runtime Linear automation)

Four Linear board sweeps drive features from idea -> shipped on the **Codex** team's **Linear Sweep** project, with full QA before anything reaches production and human sign-off on the normal path. They are **canonical, cross-runtime skills**: the same `SKILL.md` files run under Claude Code (discovered natively) and Codex (via this section). Read the whole `SKILL.md` before acting.

## Coding workflow

- For any task involving writing, reviewing, debugging, refactoring, or otherwise changing code, invoke the installed Andrej Karpathy skill (`andrej-karpathy-skill` from the `andrej-karpathy-skills` plugin) before starting that work.
- If the skill is unavailable in the current session, apply its core checks manually: think before coding, keep the change simple, make surgical edits, and verify the goal before calling the work complete.

## Delegation preference

- By default, delegate independent, bounded research, implementation, verification, or review work to subagents when parallel execution is safe and useful. Keep shared-state edits coordinated by the primary agent, and do not delegate when the work is too small or tightly sequential for delegation to help.

**Config:** all four read `.claude/linear-sweep.json` at the repo root for the team/project/prefix and repo-specific paths - nothing is hardcoded. Team `Codex` (key `COD`), project `Linear Sweep`, issue prefix `COD-###`. Deploy path: No production app deploy for this kit. Shipping is merge/push to main; any release publishing or external distribution step must be attended by the owner or tracked as a Todo card. Optional `repoRouting.byLabel` selects one primary repo per card and fails closed unless exactly one mapped label is live; scheduled children recheck that route before material work. Optional `reviewLenses` gate card-type review lenses. Commit-bound QA-to-Ship automatic routing applies only after full QA when the final origin SHA exactly matches Dev's reviewed SHA; `requireShipApproval: true` always preserves `Signoff`. `qa-sweep` never merges or deploys, and this queue transition adds no immediate launcher handoff to the single-runner Ship stage. `parallel.maxNonShipDispatches` counts non-Ship workspace/sweep candidates; `parallel.sameRepoCardLimits` counts per-primary-repo child slots inside a selected non-Ship candidate. Ship is limited to one child per registered source workspace, may coexist with other stages, and refills that workspace slot immediately after completion.

**Repo scope:** default to one deployable repo per card. If a feature spans sibling repos, split it into per-repo implementation cards under a parent/product card, or make the workspace explicitly multi-repo: every touched repo must be in `config.repos`, and `config.deploy` must describe every production target and canary. Do not let an unconfigured sibling-repo branch reach this repo's Ship path.

**Dependency semantics:** scheduled children must run `node "$AUTO_SWEEP_KIT_PATH/scripts/linear.mjs" dependency-status "$AUTO_SWEEP_ISSUE"` before their first material mutation and fail closed unless every blocker is in exact canonical `Done`. For an independently completable blocker: (1) search for the stable `[auto-sweep-dependency <dependent> blocked-by <blocker>]` audit marker and matching/orphaned issue, (2) create or reuse the blocker issue, (3) create the `blockedBy` relation only if absent, (4) add the marker comment only if absent, and (5) re-read the relation, stop material work, and release only the dependent's owned claim. Use the relation alone: never add `blocked:needs-user` merely because a `blockedBy` relation exists. A direct human answer without its own issue retains the existing human-block label path.

**Claim ownership:** the launcher exports separate `AUTO_SWEEP_OWNER_TOKEN` and `AUTO_SWEEP_CLAIM_DECLARATION` values. Each epoch starts with `[auto-sweep-claim v1 claim=<claim> owner=<owner> declaration=<declaration>]`; complete Linear comment history resolves ownership by first-declaration-wins. `[auto-sweep-heartbeat v1 claim=<claim> declaration=<declaration> at=<ISO>]` is liveness only, never ownership. Every release or terminal mutation complete-reads and proves both tokens, posts and verifies `[auto-sweep-claim-close v1 ...]`, then mutates the label/state. Run `node scripts/linear-watch.mjs claim-migration-status --json` before rollout. An attended `claim-migration-reset <anchor> <Issue> <Claim> <Target>` accepts only `legacy-unowned` with target `legacy`, or `orphan-declaration` with the exact declaration ID. For `ambiguous`, inspect and correct/remove the exact malformed or conflicting Linear marker, then rerun `claim-migration-status`; reset never repairs ambiguity.

**Codex tool mapping (all four):** read/search/git via `shell`; edit files via `apply_patch`; subagents via `spawn_agent`/`wait_agent`/`close_agent` (needs `[features] multi_agent = true` in `~/.codex/config.toml`); progress via `update_plan`; run the dev server via `shell` in the background (no `preview_*` tools on Codex). Use your own commit attribution - not the `Co-Authored-By: Claude` trailer shown in the files. Detect worktree/branch state with read-only git before creating/finishing branches.

- **Spec sweep** - "Spec" -> "Dev" (docs-only, + card-type-gated design/devex/security plan reviews): `.claude/skills/spec-sweep/SKILL.md`. Engineering review runs in prose mode (auto-decide via prose; never AskUserQuestion, never BLOCKED).
- **Dev sweep** - "Dev" -> "QA" (writes code on a worktree, claims active work with `dev:in-progress`, gated security/perf/design lenses, code review + an independent reviewer subagent, verifies green, pushes the branch, no merge): `.claude/skills/dev-sweep/SKILL.md`.
- **QA sweep** - "QA" -> "Signoff" normally, or "QA" -> "Ship" for an eligible unchanged fast path (smoke-test as a user via `/qa`, `/design-review`, fix UX bugs, screenshots to the card; **never merges, never deploys**): `.claude/skills/qa-sweep/SKILL.md`.
- **Ship sweep** - "Ship" -> "Done" (the only sweep that merges + deploys; canary-verifies; resume-on-merge-commit): `.claude/skills/ship-sweep/SKILL.md`. **Single-runner** (pin dispatch to one host) and accepts human-approved cards or commit-bound fast paths promoted by qa-sweep.
- **Unblock sweep** - manual-only interactive review of cards carrying `blocked:open-questions`, `blocked:needs-user`, `qa:needs-changes`, or `sweep:manual-only` across registered anchors: `.claude/skills/unblock-sweep/SKILL.md`. Never scheduled; clears blockers only after a human supplies a resolution.

**Statuses:** Normal path: Backlog -> Spec -> Dev -> QA -> **Signoff** -> **Ship** (human move) -> Done. Eligible unchanged fast path after full QA: QA -> **Ship** automatically; `requireShipApproval: true` disables that branch and preserves Signoff. Active development is `Dev` plus `dev:in-progress`; the old `In Progress` state is legacy cleanup only. Plus **`Todo`** = a human-only action item the agent can't do (DNS, a dashboard secret, a third-party console step, a deploy the agent can't trigger); sweeps spin these off and link them to the feature card so the user has one tracked to-do list. Also Canceled/Duplicate/Archived. **Workflow labels:** `spec:in-progress`, `dev:in-progress`, `qa:in-progress`, `qa:needs-changes`, `qa:passed`, `ship:in-progress`, `ship:approved`, `fast-path:eligible`, `blocked:open-questions`, `blocked:needs-user`, `sweep:manual-only`.

**Auth:** `LINEAR_API_KEY` lives in the gitignored `.env` at repo root (`set -a && . ./.env && set +a`), never committed.

## Factory Learning Loop

Factory Learning observes bounded structured evidence through three lenses: reliability, quality/rework, and throughput/cost. A single registry-pinned learning runner executes only after delivery work drains and never receives repository write tools or secret-bearing environment values. Medium- and high-confidence findings automatically create or update `factory:learning-generated` cards at the bottom of Spec; low-confidence patterns accumulate without creating cards.

Generated cards follow Spec -> Dev -> QA -> Signoff and always require the human Ship move. They are never fast-path eligible, and Ship requires `qa:passed`. A `factory:learning-generated` card never auto-ships or uses the auto-ship marker and always requires the human move to Ship. Use `node scripts/linear-watch.mjs learning-status --json` for read-only diagnostics and `node scripts/linear-watch.mjs learning-run --dry-run` to preview deterministic create/update/evaluation decisions without Linear writes or cursor movement. Disable the workspace `learning.enabled` setting or the registry learning runner as the kill switch. Automatic recurrence is capped at three generations before `blocked:needs-user` human review.

## Linear feature tracking (going forward)

Track every product/engineering feature, bug, or product-impacting change on the Linear Sweep board. Find or create a `COD-*` issue for the work itself (not meta "design X" cards - attach specs/plans/review notes to the feature card). Put the `COD-###` key in the branch name / commit subjects where practical. Cards created or moved during direct user conversations, or by any non-sweep skill, should carry `sweep:manual-only` unless the user explicitly wants the unattended sweeps to pick them up immediately; clear it only when handing the card back to the normal sweep pipeline. Raw ideas -> Backlog; selected-but-underspecified -> Spec; designed/active -> Dev (active work carries `dev:in-progress`); PR/review -> QA; normally QA-passed -> Signoff -> human-approved Ship, while an unchanged commit-bound fast path may move from QA directly to Ship after full QA; shipped/verified -> Done. Work discovered after the fact -> a `Done` card titled `Completed: ...` with a short summary + evidence.

Default to one deployable repo per card. True multi-repo cards require every touched repo in `config.repos` plus deploy/canary instructions for every production target; otherwise split into per-repo child cards before Dev/QA/Ship.
